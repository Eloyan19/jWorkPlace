# FAQ jWorkPlace — поддержка пользователей

Ответы на частые вопросы о продукте jWorkPlace. Ассистент поддержки отвечает СТРОГО по этому
документу; если ответа здесь нет — эскалирует запрос человеку.

## Что такое jWorkPlace

jWorkPlace — веб-сервис, который изучает произвольный GitHub-репозиторий по ссылке и ассистирует
по нему: отвечает на вопросы о коде (grounded-чат с цитатами), показывает структуру проекта,
предлагает правки и открывает Pull Request. Работает поверх RAG-индексации кода (FAISS + локальные
эмбеддинги nomic-embed) и облачной LLM DeepSeek за провайдер-абстракцией.

## Как подключить репозиторий

Вставьте ссылку на публичный GitHub-репозиторий в панели «Проекты» и нажмите «Подключить». Сервис
склонирует репозиторий (shallow-клон, без исполнения кода), просканирует его на секреты, разобьёт
код на чанки по символам и построит поисковый индекс. Когда статус проекта станет «готов», можно
задавать вопросы в чате и переключаться между проектами.

## Почему репозиторий долго индексируется

Скорость индексации зависит от размера репозитория и числа файлов. Основное время уходит на
вычисление эмбеддингов через локальную модель nomic-embed на CPU (сервер работает на VPS с 3.8 ГБ
RAM, без GPU). Крупный репозиторий с тысячами файлов может индексироваться несколько минут. Ускоряют
процесс: кэш эмбеддингов (повторные и одинаковые файлы между проектами и форками не пересчитываются)
и отсев больших, бинарных и vendored-файлов до эмбеддинга. Есть потолки: не более 5000 файлов, размер
репозитория до 200 МБ, таймаут индексации 300 секунд. Если репозиторий упирается в лимит или таймаут,
индексация завершится ошибкой — уменьшите объём или переиндексируйте позже. Инкрементальная
переиндексация («Обновить») пересчитывает только изменённые файлы и проходит быстрее полной.

## Проект завис в статусе индексации

Если сервис перезапустился во время индексации, проект помечается ошибкой («прервано рестартом
сервиса») — это ожидаемое поведение, индексация не переживает рестарт. Нажмите «Переиндексировать
заново», чтобы собрать индекс с нуля. Если проект долго висит в статусе «индексируется» без рестарта —
он всё ещё обрабатывается; дождитесь перехода в «готов» или ошибку.

## Почему ассистент отвечает «не знаю»

Ответы обоснованы кодом: если ни один фрагмент кода не проходит порог релевантности, ассистент
честно отвечает «не знаю», а не выдумывает ответ из общих знаний модели. Это защита от галлюцинаций.
Переформулируйте вопрос ближе к терминам кода (имена функций, классов, файлов) — гибридный поиск
(лексический + семантический) лучше находит точные идентификаторы.

## Как включить правки и открыть Pull Request

По умолчанию проект работает в режиме «только чтение»: чат, поиск, структура и предпросмотр правки
(diff) доступны, но реальный PR — нет. Чтобы включить правки, привяжите к проекту GitHub
fine-grained Personal Access Token с правами contents:write и pull_requests:write на нужный
репозиторий (панель «Проекты», поле токена). Токен шифруется на сервере, никогда не возвращается
в браузер, не попадает в логи и в контекст LLM. После этого в панели «Правка» появится кнопка
«Подтвердить и открыть PR»: сервис создаёт ветку, коммит и Pull Request. Прямой push в основную
ветку невозможен — всегда PR из отдельной ветки, только после вашего подтверждения.

## Безопасны ли мои токены и ключи

Да. GitHub-токен хранится в зашифрованном виде (Fernet) в данных сервиса и привязан к репозиторию
своего проекта. Ключ LLM (DeepSeek) хранится только в переменных окружения сервера. Ни токен, ни
ключ никогда не отправляются в браузер, не пишутся в логи и не попадают в промпт модели. Браузер
общается только со своим backend; напрямую в GitHub или LLM он не ходит.

## Исполняет ли сервис мой код

Нет. jWorkPlace только читает файлы репозитория и выполняет git-операции (клон, ветка, коммит, PR).
Чужой код не собирается, не запускается и не тестируется. Git-хуки чужого репозитория отключены при
клонировании. Исполнение кода в изоляции — задача будущей фазы, в текущей версии его нет.

## Что такое AI-ревью Pull Request

Для проекта с настроенным GitHub Action jWorkPlace может автоматически ревьюить Pull Request'ы:
Action вызывает backend, тот анализирует diff вместе с RAG-контекстом проекта через DeepSeek и
оставляет один комментарий с замечаниями. Комментарий обновляется при новых пушах в PR.

## Поддерживаемые языки

Индексатор разбивает код по синтаксическому дереву (tree-sitter) для основных языков; для остальных
и неизвестных расширений работает разбиение по строкам с перекрытием. Один общий эмбеддер
nomic-embed используется для всех языков, а точные идентификаторы дополнительно ловит лексический
канал поиска.

## Как переключаться между проектами

Проиндексированные проекты копятся в панели «Проекты». Выбор проекта делает его активным; чат, поиск,
структура и правки работают только по активному проекту — контексты проектов не смешиваются. Удаление
проекта чистит его клон, индекс и историю; глобальный кэш эмбеддингов при этом сохраняется.
